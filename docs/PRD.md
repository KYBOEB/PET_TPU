# PRD — Питомец ТПУ 🐶

## 1. Описание продукта

**Питомец ТПУ** — веб-приложение (PWA-ready), которое превращает выполнение учебных задач в игру. У каждого студента есть виртуальный питомец-корги, который «растёт» и «питается» за счёт выполненных заданий. Нейросеть оценивает сложность задачи и назначает награду.

**Целевая аудитория:** студенты-первокурсники ТПУ.

**Платформа:** веб-браузер (мобильный + десктоп).

**Технический стек:** FastAPI + SQLAlchemy + SQLite / HTML+CSS+JS / Qwen (LLM).

---

## 2. Основные сущности

### 2.1 Пользователь (User)
- Email (валидация формата, без проверки существования)
- Пароль (хэширование bcrypt, минимум 6 символов)
- У каждого пользователя ровно ОДИН питомец (создаётся автоматически при регистрации)

### 2.2 Питомец (Pet)
- **Имя:** по умолчанию «Студент-корги», можно изменить
- **Уровень:** 0–3 (начальный — 0)
- **Опыт (EXP):** числовое значение, сбрасывается при переходе на новый уровень
- **Голод (Hunger):** 0–100 (100 = сытый, 0 = голодный)
- **Настроение (Mood):** вычисляется из голода (не хранится отдельно)

### 2.3 Задача (Task)
- **Заголовок:** текст (обязательное, макс. 200 символов)
- **Категория:** текст-маркер (обязательное, макс. 50 символов, например «Математика»)
- **Срок выполнения:** дата в формате ДД.ММ.ГГГГ (обязательное)
- **Сложность (difficulty):** low / medium / high (заполняется LLM при активации)
- **Награда EXP:** числовое значение (заполняется LLM при активации)
- **Награда Hunger:** числовое значение (заполняется LLM при активации)
- **Статус активации:** активирована / не активирована
- **Статус выполнения:** выполнена / не выполнена
- **Статус просрочки:** просрочена / не просрочена (is_overdue)

## 3. Ключевые механики

### 3.1 Система опыта и уровней

| Переход | Необходимый EXP |
|---|---|
| Уровень 0 → 1 | 200 |
| Уровень 1 → 2 | 300 (200 × 1.5) |
| Уровень 2 → 3 | 450 (300 × 1.5) |
| После уровня 3 | EXP копится, уровень не растёт |

**Формула:** `required_exp(level) = 200 * (1.5 ^ level)` (для level 0 → 200, level 1 → 300, level 2 → 450).

**При переходе на новый уровень:** текущий EXP сбрасывается в 0. Излишек НЕ переносится.

### 3.2 Система голода

- **Максимум:** 100 единиц
- **Убывание:** линейное по реальному времени. 100 единиц → 0 за 48 часов
- **Скорость убывания:** ≈ 2.083 единицы в час (100 / 48)
- **Пополнение:** при выполнении задачи hunger восстанавливается на hunger_reward задачи (не выше 100)
- **Расчёт:** при каждом обращении к серверу (GET /api/pet/status) hunger пересчитывается по формуле:
  ```
  elapsed_hours = (now - last_hunger_update).total_seconds() / 3600
  hunger = max(0, stored_hunger - elapsed_hours * (100 / 48))
  ```

### 3.3 Настроение питомца (Mood)

Вычисляется из текущего значения голода:

| Голод | Настроение | Картинка |
|---|---|---|
| 0–20 | Грустный (sad) | `pet_level{N}_sad.png` |
| 21–80 | Спокойный (neutral) | `pet_level{N}_neutral.png` |
| 81–100 | Радостный (happy) | `pet_level{N}_happy.png` |

### 3.4 Штраф за голод = 0

Когда голод достигает 0:
- Каждый реальный час у питомца отнимается **10 EXP**
- EXP не может уйти ниже 0
- Уровень НЕ понижается (уровень остаётся, но прогресс может обнулиться)
- Штраф рассчитывается при обращении к серверу аналогично голоду:
  ```
  if hunger == 0:
      penalty_hours = (now - time_when_hunger_reached_zero).total_seconds() / 3600
      exp_penalty = floor(penalty_hours) * 10
  ```

### 3.5 Оценка задач через LLM

**Триггер:** пользователь нажимает кнопку «Активировать задания».

**Процесс:**
1. Собираются все задачи пользователя со статусом `is_activated = false`
2. Отправляется один запрос к LLM со списком заголовков задач
3. LLM возвращает для каждой задачи: difficulty + exp_reward + hunger_reward
4. Значения записываются в БД, статус задач меняется на `is_activated = true`

**Диапазоны наград:**

| Сложность | EXP | Hunger |
|---|---|---|
| low | 10–20 | 5–15 |
| medium | 25–40 | 20–35 |
| high | 60–80 | 50–70 |

**Fallback (LLM недоступен):** difficulty = medium, exp_reward = 25, hunger_reward = 20.

### 3.6 Бонус за серию (Streak)

- Если пользователь выполняет **более 3 задач за последние 24 часа**, на каждое следующее задание применяется множитель **× 1.25** к EXP и Hunger наградам
- Считаются только задачи, выполненные за скользящее окно 24 часа от текущего момента
- Множитель применяется к уже назначенным наградам, результат округляется вниз

**Пример:** задача дала бы 40 exp → со стриком: floor(40 × 1.25) = 50 exp.

### 3.7 Штраф за пропуск дедлайна

Если текущая дата **больше** даты срока выполнения задачи (deadline), а задача активирована и не выполнена — применяется штраф:

- **EXP:** вычитается `exp_reward` задачи из текущего EXP питомца
- **Hunger:** вычитается `hunger_reward` задачи из текущего голода питомца
- EXP не опускается ниже 0, hunger не опускается ниже 0
- Уровень НЕ понижается
- Штраф применяется **один раз** — задача помечается как `is_overdue = true`
- Просроченная задача **исчезает из списка** (аналогично выполненной)

**Когда проверяется:** при каждом обращении к GET /api/tasks и GET /api/pet — сервер проверяет все активированные невыполненные задачи пользователя и применяет штраф к просроченным.

**Пример:** задача «Сдать лабу» (difficulty=high, exp_reward=70, hunger_reward=60) с дедлайном 10.04.2026. Сегодня 11.04.2026 → у питомца вычитается 70 exp и 60 hunger, задача помечается как просроченная и исчезает.

**Неактивированные задачи:** штраф НЕ применяется к задачам, которые не были активированы (у них нет назначенной награды). Такие задачи просто остаются в списке.

## 4. Пользовательские сценарии (User Flows)

### 4.1 Регистрация и вход
1. Пользователь открывает приложение → видит страницу входа
2. Нажимает «Зарегистрироваться» → вводит email + пароль (×2 для подтверждения)
3. При успехе: автоматически создаётся питомец «Студент-корги» (уровень 0, hunger 100, exp 0)
4. Пользователь автоматически входит в систему
5. При повторном визите: вводит email + пароль → получает JWT-токен

### 4.2 Главный экран (питомец)
1. Отображается картинка питомца (уровень + настроение)
2. Над картинкой — имя питомца + кнопка «карандаш» для редактирования
3. Под картинкой — полоска EXP (текущий / необходимый для уровня)
4. Отображается текущий уровень
5. Отображается индикатор голода (числовое или графическое)

### 4.3 Создание задачи
1. На странице задач пользователь нажимает «+»
2. Открывается модальное окно / форма:
   - Поле «Заголовок задачи»
   - Поле «Категория»
   - Поле «Срок выполнения» (DD.MM.YYYY)
3. Нажимает «Создать» → задача появляется в списке
4. Задача ещё НЕ активирована (награда не назначена)

### 4.4 Активация задач
1. Пользователь нажимает «Активировать задания»
2. Все неактивированные задачи отправляются на оценку LLM
3. После ответа — у каждой задачи появляется награда
4. В списке отображается: заголовок, срок, награда (EXP + hunger)

### 4.5 Выполнение задачи
1. Пользователь нажимает чекбокс (квадрат) слева от задачи
2. Начисляются EXP и Hunger (с учётом стрик-множителя)
3. Если EXP достаточно для нового уровня — уровень повышается, EXP сбрасывается
4. Задача исчезает из списка

### 4.6 Редактирование задачи
1. Пользователь нажимает «карандаш» рядом с задачей
2. Открывается форма с текущими значениями (заголовок, категория, срок)
3. Внизу — красная кнопка «Удалить задание»
4. При редактировании активированной задачи: награда сбрасывается, задача возвращается в статус «не активирована» (требуется повторная активация)

### 4.7 Страница чата (заглушка)
1. Отображается сообщение от питомца:
   > «Извините! Чат ещё в разработке, но скоро мы сможем пообщаться!
   > Если у тебя появились вопросы, касающиеся обучения в университете, попробуй найти ответ здесь.»
2. Под сообщением — кнопка-ссылка на help.tpu.ru

## 5. Edge Cases

| Ситуация | Поведение |
|---|---|
| Hunger > 100 после выполнения задачи | Обрезается до 100 |
| EXP < 0 после штрафа | Обрезается до 0, уровень НЕ понижается |
| Уровень = 3, EXP набрано на переход | Уровень остаётся 3, EXP копится дальше |
| LLM вернул невалидный ответ | Применяется fallback: medium / 25 exp / 20 hunger |
| LLM вернул значения вне диапазона | Значения клампятся к границам диапазона |
| Пользователь нажал «Активировать» без новых задач | Ничего не происходит, сообщение «Нет новых задач для активации» |
| Пользователь пытается выполнить неактивированную задачу | Запрещено: чекбокс неактивен или скрыт для неактивированных задач |
| Задача отредактирована после активации | Награда сбрасывается, нужна повторная активация |
| Два пользователя с одинаковым email | Регистрация отклоняется: «Email уже зарегистрирован» |
| JWT-токен истёк | Статус 401, фронтенд редиректит на страницу входа |
| Пользователь не заходил 7 дней | Голод = 0 давно, штраф за EXP ограничен минимумом 0 |
| Задача просрочена, но не активирована | Штраф НЕ применяется (нет назначенной награды) |
| Задача просрочена и уже выполнена | Штраф НЕ применяется (задача завершена вовремя) |
| Несколько задач просрочены одновременно | Штраф за каждую применяется последовательно, EXP и hunger не ниже 0 |
| Hunger < 0 после штрафа за дедлайн | Обрезается до 0 |

## 6. Требования к картинкам питомца

### Для дизайнера (Егор):

- **Количество:** 9 штук (3 уровня × 3 настроения)
- **Формат:** PNG с прозрачным фоном
- **Разрешение:** 512 × 512 пикселей
- **Цветовая схема:** в рамках общей палитры проекта (бежево-жёлто-оранжевая)
- **Стиль:** мультяшный корги, единообразный стиль между уровнями

**Имена файлов (строго):**

| Файл | Описание |
|---|---|
| `pet_level0_sad.png` | Уровень 0, грустный |
| `pet_level0_neutral.png` | Уровень 0, спокойный |
| `pet_level0_happy.png` | Уровень 0, радостный |
| `pet_level1_sad.png` | Уровень 1, грустный |
| `pet_level1_neutral.png` | Уровень 1, спокойный |
| `pet_level1_happy.png` | Уровень 1, радостный |
| `pet_level2_sad.png` | Уровень 2, грустный |
| `pet_level2_neutral.png` | Уровень 2, спокойный |
| `pet_level2_happy.png` | Уровень 2, радостный |

> Дополнительно для уровня 3:
> `pet_level3_sad.png`, `pet_level3_neutral.png`, `pet_level3_happy.png`
> Итого: **12 картинок** (4 уровня: 0, 1, 2, 3).

**Различия между уровнями:** питомец должен визуально «расти» или меняться (например, щенок → подросток → взрослый → корги в академической шапочке).

**Заглушки для разработки:** простые цветные круги с текстом уровня и настроения (генерируются программно на этапе разработки).

## 8. Нефункциональные требования

- **Адаптивность:** корректное отображение на экранах 320px–1440px
- **Время отклика:** API < 500ms (без учёта LLM-запросов)
- **LLM-запрос:** таймаут 30 секунд, при превышении — fallback
- **Безопасность:** bcrypt для паролей, JWT (срок жизни — 7 дней), rate limiting на /auth эндпоинтах
- **Логирование:** JSON-формат, LLM-запросы/ответы логируются с маскировкой PII
